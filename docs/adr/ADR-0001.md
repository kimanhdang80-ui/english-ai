# ADR-0001 — Next.js full-stack + Supabase Auth for the MVP

- **Status:** Accepted
- **Date:** 2026-07-01
- **Deciders:** Lead Architect
- **Sprint:** 1 (reaffirmed & standardized in Sprint 1.2)
- **Supersedes:** the NestJS-monorepo assumption in the original
  [SYSTEM_ARCHITECTURE.md](../SYSTEM_ARCHITECTURE.md) (for the MVP window only)

## Context

The documentation set ([SYSTEM_ARCHITECTURE.md](../SYSTEM_ARCHITECTURE.md))
originally assumed a **pnpm/Turborepo monorepo** with a separate **NestJS** backend,
**custom JWT** auth, Redis/BullMQ workers, and Kubernetes deployment.

The Sprint 1.1 build directive instead specified **Next.js (App Router)** as the
application framework, **Supabase Auth**, **Prisma + PostgreSQL**, and deployment to
**Vercel** (web) + **Railway** (container). We need a foundation that ships an MVP
quickly **without** foreclosing the long-term, scale-oriented architecture the docs
describe. This ADR records how those two are reconciled.

## Decision

For the MVP we run a **single Next.js full-stack application**:

- **UI + API in one Next.js app** — Server Components, Route Handlers, and Server
  Actions replace the separate NestJS service. Domain logic lives in framework-light
  modules (`src/lib`, later `src/modules` / extracted `packages/*`) so it stays
  testable and portable.
- **Supabase Auth** for authentication (managed sessions, OAuth, email flows) via the
  `@supabase/ssr` clients in `src/lib/supabase`.
- **Prisma + PostgreSQL** for data (Supabase provides Postgres in production).
- **Deploy:** Vercel for the web app; Railway (via Dockerfile / standalone output) as
  a container host and the future home of the background **worker**.

Clean-architecture guarantees are kept regardless of packaging: provider
abstraction (auth/AI/storage behind modules, never vendor SDKs in UI), strict
TypeScript, and inward-pointing dependencies.

## Alternatives

### A. Full monorepo now — `apps/web` (Next.js) + `apps/api` (NestJS) + `packages/*`

- **Pros:** hard service boundary from day one; independent scaling/deploy of API;
  closest to the long-term target; clear ownership seams.
- **Cons:** two runtimes/deploys to operate, cross-service auth/session plumbing, more
  CI/infra, slower to a shippable MVP — cost paid **before** there is scale to justify
  it. Rejected as premature for the MVP.

### B. Single Next.js app, single deploy _(chosen)_

- **Pros:** one deployable, one language boundary, managed auth, fastest path to
  Roadmap milestone M1; Vercel-native.
- **Cons:** long-running/async and realtime workloads don't fit Next's request model
  and will need extraction later; risk of business logic leaking into route
  handlers/components if discipline slips (mitigated by CLAUDE.md rules + reserved
  `packages/*` scaffolding).

### C. Serverless functions + separate SPA

- **Pros:** granular scaling.
- **Cons:** more moving parts than B with none of A's clean domain boundary; poor DX
  for our team size. Rejected.

## Consequences

**Positive**

- Shippable core sooner; minimal ops burden; managed auth reduces security surface.
- Domain code written framework-light is relocatable into `packages/*` without rewrite.

**Negative / Risks**

- Heavy async AI work, media/TTS, and realtime (WebSocket) do **not** belong in the
  Next request lifecycle → they become the first extraction (`apps/worker`).
- Discipline required to keep logic out of route handlers/components.

**Mitigations**

- Reserved monorepo scaffolding (`apps/`, `packages/`) created in Sprint 1.2 so
  extraction is a **move**, not a redesign — see
  [ARCHITECTURE_EVOLUTION.md](../ARCHITECTURE_EVOLUTION.md).
- Deterministic domains (`learning-engine`, `srs`) and provider ports (`ai`,
  `database`) are authored as if already packaged.

## Future Review

Re-evaluate this decision when **any** trigger below is met (details and metrics in
[ARCHITECTURE_EVOLUTION.md](../ARCHITECTURE_EVOLUTION.md)):

- We introduce durable background jobs (AI lesson generation, TTS rendering,
  scoring, SRS batch) → **extract `apps/worker`** (BullMQ + Redis). _Expected: Sprint 17–22._
- We add realtime conversation/speaking (WebSocket) that Next/Vercel can't serve well
  → **extract a realtime/API service**.
- API p95 latency, cost, or deploy-coupling between web and API becomes a measured
  bottleneck.
- A second client (Expo mobile) needs to share typed domain code → **populate
  `packages/*`** in earnest.

**Scheduled checkpoint:** end of Phase 2 (Sprint 24). If no trigger has fired, record
"still single-app" and set the next checkpoint.
