# ADR-0003 — Real AI provider pattern + `ai_usage_logs` telemetry

- **Status:** Accepted
- **Date:** 2026-07-01
- **Deciders:** Senior AI Engineer
- **Sprint:** Milestone 1 (Real AI) / Sprint 7.2
- **Relates to:** [ADR-0001](./ADR-0001.md) (single Next.js app), [AI_ENGINE.md](../AI_ENGINE.md),
  Sprint 7.1 foundation (DECISIONS D-0022…D-0024)

## Context

Sprint 7.1 shipped the AI foundation: a provider-agnostic `LlmPort`, versioned
prompt templates (prompts-as-data), renderer, validators, and a `StubLlmAdapter`
that throws (→ 501). Three learner-facing capabilities used **mocks / stubs**:
vocabulary explanation (rule-based `MockExplanationAdapter` in `daily-loop`),
example generation, and short-answer feedback (foundation-only).

Milestone 1 replaces the mocks with **real AI** while honoring the constraints:
_no architecture change, no framework expansion, no UI change, no Learning-Engine
change, and no DB change beyond what is required to store AI history._

Two decisions need recording: (1) the **provider pattern** (how vendors are
abstracted, selected, and made resilient), and (2) the **one schema change** —
an append-only `ai_usage_logs` table — which triggers this DB gate (PROJECT_OS §4).

## Decision

### 1. Provider pattern (no new architecture — adapters behind the existing port)

- `AIProvider` is the vendor seam (structurally the existing `LlmPort`). Concrete
  adapters `ClaudeProvider` (Anthropic Messages API) and `OpenAIProvider` (Chat
  Completions) call the vendor **REST APIs via `fetch`** — no vendor SDK dependency.
- `ProviderFactory.create(config)` builds the chain **from configuration** (env →
  `aiEnv`): pick provider, wrap with `RetryingProvider` (retry + timeout), and, if a
  fallback provider is configured, wrap in `FallbackProvider`. Switching providers is a
  one-line env change (`AI_PROVIDER`), never a code edit. Gemini later = one more adapter.
- **Resilience:** timeout via `AbortController` (`AI_TIMEOUT_MS`); exponential-backoff
  retry on retryable errors only (429/5xx/timeout/network) (`AI_MAX_RETRIES`);
  provider-level fallback; **capability-level fallback** to deterministic text so the
  learning loop never breaks; and structured usage logging on every call.
- **Keys come only from the environment** (`ANTHROPIC_API_KEY` / `OPENAI_API_KEY`),
  never hardcoded and never logged (CLAUDE.md §9).

### 2. `ai_usage_logs` (the single DB change — append-only telemetry)

Add the table already specified in [DATABASE.md](../DATABASE.md) §AI:
`id, user_id (nullable FK → profiles), feature, provider, model, tokens_in,
tokens_out, cost_micro_usd, latency_ms, cache_hit, occurred_at`, indexed on
`(occurred_at)`, `(feature)`, `(model)`. Writes go through an `AiUsageLogRepository`
port: a Prisma implementation when a DB is configured, a **no-op** otherwise (AI still
runs without persistence). Logging never throws into the call path.

## Impact Analysis

- **Tables:** +1 (`ai_usage_logs`). One nullable FK to `profiles` (+ one back-relation
  field on `Profile`). No change to any existing column or row.
- **Modules:** `src/modules/ai` (providers, usage-log repo, `AiTextService`); one new
  adapter in `daily-loop` (`AiExplanationAdapter`) swapped in at its container. No change
  to Learning Engine, vocabulary domain, or presentation/UI.
- **APIs:** none added or changed (no API gate needed). Capabilities are reachable via the
  `ai` container's services; UI wiring is a later sprint.
- **Data volume:** append-only, one row per AI call. Hot table → already earmarked for
  monthly range-partitioning + rolling archival in DATABASE.md §scale (not needed at MVP).
- **Read/write paths:** write-only on the AI path (fire-and-forget, failure-swallowed);
  reads are future cost dashboards only.

## Migration Plan

- **Expand-only, additive, backward-compatible.** New table + indexes; a nullable FK.
  Forward migration `prisma/migrations/20260701010000_ai_usage_logs/migration.sql`
  (generated via `prisma migrate diff`). No backfill (append-only from first write).
- Apply order: create table → add indexes → add FK. `prisma migrate deploy` on a live DB.
- **Environment note:** no live Postgres in the build environment, so the migration is
  authored and validated (`prisma validate` + `generate`) but **applied when the DB is
  provisioned** (tracked with DEBT-004). Until then the no-op repo is used.

## Rollback Plan

- Fully reversible with **no data at risk in existing tables** (only the new table holds
  data): `DROP TABLE ai_usage_logs;` (drops its indexes/FK with it). Because writes are
  fire-and-forget and the repo is swappable, reverting to the no-op repo disables logging
  instantly without a schema change if needed.

## Alternatives

### A. Use the vendor SDKs (`@anthropic-ai/sdk`, `openai`)

Rejected for now — adds two dependencies (bundle + supply-chain + version churn) for what
is a single REST call each. `fetch` gives full control of timeout/abort and keeps the
adapter tiny and dependency-free. Revisit if we need streaming/tool-use ergonomics.

### B. Log usage in the provider decorator (a `LoggingProvider`)

Rejected — the provider layer doesn't know the **capability** (`feature`) or **userId**.
Logging lives in the capability services (`AiTextService`), which have that context; the
provider stays a pure transport.

### C. Reuse `ai_generation_jobs` / skip persistence

Rejected — `ai_usage_logs` is the DATABASE.md-specified home for per-call cost/latency;
`ai_generation_jobs` models async job state, a different concern. Skipping persistence
would violate the milestone's "store AI history" requirement.

## Consequences

**Positive**

- Real Claude/OpenAI behind the same port; provider swap = env change; resilient by
  default; every call is observable (cost/latency/outcome).
- The learning loop degrades gracefully to deterministic text when AI is off or failing.

**Negative / Risks**

- The migration is **authored but not yet applied** here (no DB) — same standing debt as
  the rest of the schema (DEBT-004); AI history is not persisted until then (no-op repo).
- Real provider quality/latency/cost are unverified without keys + network (RISK); an
  evaluation harness (AI_ENGINE §6) is future work.

## Future Review

- When keys + a DB exist: apply the migration, enable the Prisma repo, and validate output
  quality with a golden-dataset harness; add per-user AI quotas at the API layer.
- Add a dedicated `ai.generate` permission when generation is surfaced in the UI (DEBT-015).
- Revisit vendor SDKs if streaming/tool-use is needed; add Gemini as a third adapter.
