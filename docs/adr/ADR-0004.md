# ADR-0004 — Supabase→app identity sync via database triggers

- **Status:** Accepted
- **Date:** 2026-07-02
- **Deciders:** Principal Backend Engineer
- **Task:** RC-02 (Auth Production Ready)
- **Relates to:** [ADR-0002](./ADR-0002.md) (Supabase-Auth-centric identity), DEBT-008,
  PRODUCTION_AUDIT PA-C2

## Context

ADR-0002 established that **Supabase owns `auth.users`** and our `public` schema owns application
identity: `profiles.id EQUALS auth.users.id`, with **no cross-schema FK** — the link is "by
convention + a DB trigger/webhook wired at deploy." That trigger was never created (DEBT-008).

Consequence (confirmed in code): `signUpAction` creates a Supabase user and passes
`display_name` in metadata, but **nothing creates the `profiles` row or assigns a role**. So a
verified user has `profile = null` and an **empty permission set** (`getUserAccess`), every
`requirePermission` fails, and FK inserts (`user_vocabulary`, `audit_logs`) reject. This is the #1
auth blocker to any real beta.

Also unaddressed: **account deletion**. Because there is no cross-schema FK, deleting an
`auth.users` row leaves an orphaned `profiles` row (and its owned data).

## Decision

Wire identity sync with **two `AFTER` triggers on `auth.users`** (migration
`20260702000000_auth_user_sync`):

1. **`handle_new_user()` (AFTER INSERT):** insert `public.profiles (id=NEW.id, email,
display_name from `raw_user_meta_data`)` then assign the default **`student`** role via
   `public.user_roles`. `SECURITY DEFINER`, `search_path=public`, both inserts
   `ON CONFLICT DO NOTHING` (idempotent, safe to retry / coexist with re-sends).
2. **`handle_user_delete()` (AFTER DELETE):** `DELETE FROM public.profiles WHERE id=OLD.id`, which
   **cascades** to owned rows (`user_roles`, `user_vocabulary`, `review_history`, …) and SetNulls
   audit/AI logs — a clean, GDPR-aligned account teardown at the data layer.

The default role (`student`) matches `DEFAULT_ROLE` in `src/lib/auth/permissions.ts`; roles are
seeded by `prisma/seed.ts` before sign-ups occur.

## Impact Analysis

- **Tables:** none added/changed. Two functions + two triggers on `auth.users` (Supabase schema).
- **Data paths:** one extra insert-pair per sign-up (profile + role); one cascade delete per account
  removal. Negligible volume.
- **Code:** no application code change required — `signUpAction` already sends `display_name`;
  `getCurrentUser`/`getUserAccess` already read `profiles`/`user_roles`. The trigger fills the gap
  they assumed existed.
- **APIs/UI:** none. No new endpoints, no UI, no UX change.
- **Ordering:** roles must be seeded first (they are, in `prisma:seed`); the role lookup no-ops
  safely if not.

## Migration Plan

- Forward: the raw-SQL migration above, applied by `prisma migrate deploy` (runs after the schema
  migrations). Backward-compatible (additive; existing rows untouched).
- **Backfill (deploy-time, for any users created before the trigger):**
  ```sql
  INSERT INTO public.profiles (id, email, created_at, updated_at)
    SELECT u.id, u.email, now(), now() FROM auth.users u
    LEFT JOIN public.profiles p ON p.id = u.id WHERE p.id IS NULL;
  INSERT INTO public.user_roles (id, user_id, role_id, created_at)
    SELECT gen_random_uuid(), u.id, r.id, now()
    FROM auth.users u CROSS JOIN public.roles r
    LEFT JOIN public.user_roles ur ON ur.user_id = u.id AND ur.role_id = r.id
    WHERE r.code = 'student' AND ur.id IS NULL;
  ```

## Rollback

```sql
DROP TRIGGER IF EXISTS on_auth_user_created ON auth.users;
DROP TRIGGER IF EXISTS on_auth_user_deleted ON auth.users;
DROP FUNCTION IF EXISTS public.handle_new_user();
DROP FUNCTION IF EXISTS public.handle_user_delete();
```

Fully reversible; no data in existing tables is destroyed by dropping the triggers (they only
stop future auto-sync). Profiles created while active remain.

## Alternatives

### A. Auth webhook (Edge Function / server route) instead of a DB trigger

Rejected for the core path: a trigger is transactional with the auth insert, needs no extra
deployment/secret, and can't be missed by a client that forgets to call it. (A webhook remains an
option for side effects like welcome emails.)

### B. Create the profile in `signUpAction` (application code)

Rejected as the source of truth: sign-up can happen via OAuth/magic-link/other flows that don't go
through `signUpAction`; the trigger covers every path. (App code may still upsert opportunistically.)

## Consequences

**Positive:** real sign-ups get a profile + `student` role automatically (every auth path);
account deletion cleans up cascade-owned data; no app/UI change; idempotent + reversible.

**Negative / Risks:** the trigger lives in the DB (must be applied via migration and verified in
each environment); `SECURITY DEFINER` runs as owner — scoped narrowly to the two inserts/one delete.
Cannot be runtime-verified without a live Supabase DB (this environment has none) — see the
verification matrix in AUTH_READY_REPORT.md.
