# ADR-0002 — Supabase-Auth-centric identity model + application RBAC

- **Status:** Accepted
- **Date:** 2026-07-01
- **Deciders:** Lead Architect
- **Sprint:** 2.1
- **Relates to:** [ADR-0001](./ADR-0001.md) (chose Supabase Auth)

## Context

ADR-0001 chose **Supabase Auth** for authentication. Supabase owns the `auth` schema
(`auth.users`, `auth.identities`, tokens, email verification, password hashing). The
Sprint 1 Prisma schema, however, still modeled an app-side `User` with `password_hash`
and an `AuthAccount` OAuth-linking table — duplicating what Supabase already owns.

Sprint 2.1 must build a production auth foundation and design the identity/RBAC tables
(`profiles`, `roles`, `permissions`, `role_permissions`, `user_roles`, `user_settings`,
`user_devices`, `user_sessions`, `audit_logs`, `notification_preferences`). We must
decide where the source of truth for identity lives and how authorization is modeled.
Per project rules, an architecture change needs an explicit analysis — hence this ADR.

## Decision

1. **Supabase owns authentication; our `public` schema owns application identity.**
   - `Profile.id` (uuid) **equals** `auth.users.id`. No cross-schema FK; the row is
     created/synced by a Supabase DB trigger / webhook (wired at deploy).
   - We store **no passwords** and no OAuth identities in `public` — Supabase holds them.
   - **Removed** the Sprint 1 `User`, `UserProfile`, and `AuthAccount` models; `Profile`
     is now the single app-side identity, and `Language`/content FKs point at `Profile`.
2. **`user_sessions` mirrors sessions for audit + "log out everywhere"** — it stores a
   token _hash_ + metadata only, never raw Supabase tokens.
3. **Authorization is permission-based RBAC, never role-name checks.**
   - `permissions` are atomic (`<resource>.<action>`). `roles` bundle permissions via
     `role_permissions`. Users hold roles via `user_roles`.
   - Code checks `permissions.has('content.publish')`, never `role === 'admin'`. Roles
     are seeded **data**, so they evolve without code changes.
4. **Two-layer enforcement:** Edge **middleware** checks _authentication_ only (no DB on
   Edge); Node-runtime layouts/pages check _authorization_ via `requirePermission()`.

## Alternatives

### A. Keep an app-managed `User` with password_hash (custom auth)

Rejected — contradicts ADR-0001, doubles the security surface, and creates a second
source of truth for credentials. Supabase already solves email/OAuth/verification.

### B. Store everything (including roles) inside Supabase `auth.users.app_metadata`

Rejected — RBAC needs relational joins, an auditable mapping, and admin tooling.
JSON in `app_metadata` is opaque, hard to query, and couples authz to the auth vendor.
We do plan to mirror a lightweight role claim into the JWT later for Edge checks (see
Future Review) — but the DB remains the source of truth.

### C. Role-based checks in code (`if role === 'admin'`)

Rejected explicitly by requirements ("không hard-code role") and by CLAUDE.md §9:
brittle, and every new capability would require code edits. Permission checks scale.

## Consequences

**Positive**

- One source of truth for credentials (Supabase) and one for app identity/authz (our DB).
- RBAC is data-driven, auditable, and admin-manageable; permissions decouple features
  from roles.
- Clean separation lets Edge middleware stay DB-free and fast.

**Negative / Risks**

- `Profile` creation depends on a Supabase→DB sync (trigger/webhook) — must be wired and
  monitored; until then a profile may briefly not exist (handled: code tolerates null).
- Permission checks require a DB read per protected Node render (mitigate later with a
  short-lived cache or a role claim in the JWT).

**Migration**

- Destructive at the model level, but pre-launch with **no production data**, so it is a
  schema replacement, not a data migration. Prisma generates the DDL.

## Future Review

- When Edge-level authorization is needed (e.g. gating `/admin` at the edge), mint a
  **custom JWT claim** with role/permission summary via a Supabase Auth Hook and verify
  it in middleware — revisit then.
- If permission-read latency shows up in traces, add a cached access snapshot.
- Checkpoint: end of Epic 2 (auth complete) — confirm the trigger/webhook sync is in place.
