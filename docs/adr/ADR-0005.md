# ADR-0005 — Persistence stores for the runtime (retire in-memory repositories)

- **Status:** Accepted
- **Date:** 2026-07-02
- **Deciders:** Principal Backend Engineer
- **Task:** RC-03 (Persistence Production Ready)
- **Relates to:** [ADR-0003](./ADR-0003.md) (`ai_usage_logs`), the Learning Model V2 migration
  (docs/migration/DATABASE_MIGRATION_PLAN.md, P2), PRODUCTION_AUDIT (PA-H1…H4), DEBT-014/016

## Context

Several modules shipped their infrastructure as **in-memory skeleton repositories** behind stable
ports (documented as tech debt at each step): the Daily Loop session store and lesson-plan store,
the Mission Engine mission + progress stores, and the AI prompt-template + generation-history
stores. In parallel, the Mission Library (Task 04) was authored as JSON under `content/*` and read
**directly from the filesystem at runtime** by the mission-flow UI.

Consequences: nothing survives a process restart (streak/history/progress/plans reset), horizontal
scaling is unsafe (each instance has its own memory), and content is coupled to the deploy artifact.
This is the last structural blocker before the loop can be "real and durable" (Sprint 8.2 intent).

## Decision

Introduce **durable Postgres stores** for every in-memory runtime repository and move the Mission
Library into the database, in one additive migration (`20260702010000_persistence_stores`). Ports
are unchanged; only the wiring in the composition roots swaps in-memory → Prisma.

New tables (8):

| Table                | Replaces (in-memory)                  | Shape                                                   |
| -------------------- | ------------------------------------- | ------------------------------------------------------- |
| `learning_sessions`  | `InMemorySessionRepository`           | one row per completed session; FK→profiles (cascade)    |
| `lesson_plans`       | `InMemoryLessonPlanRepository`        | one row per user (unique), plan as JSON; FK→profiles    |
| `content_tracks`     | filesystem JSON (`content/tracks`)    | track document (JSON) + indexed scalars; PK = track key |
| `content_missions`   | filesystem JSON (`content/missions`)  | mission document (JSON) + indexed scalars; FK→tracks    |
| `mission_progress`   | `InMemoryUserProgressRepository`      | unique [user, mission]; FK→profiles (cascade)           |
| `prompt_templates`   | `InMemoryPromptTemplateRepository`    | template metadata; seeded from the code registry        |
| `prompt_versions`    | `InMemoryPromptTemplateRepository`    | version body/model/budget; unique [template, version]   |
| `ai_generation_jobs` | `InMemoryGenerationHistoryRepository` | append-only generation records (request/result as JSON) |

**Content as documents.** `content_tracks`/`content_missions` store the validated `TrackContent` /
`MissionContent` in a `jsonb` column, with the query/order fields (`key`, `track_key`,
`mission_order`, `title`, `difficulty`, …) promoted to indexed scalar columns. The runtime reads
rows and **re-validates them with the same Zod schema** (`src/content/mission-schema.ts`) — the
single source of truth for shape. The authored JSON files become **seed-only** input
(`prisma/data/mission-library.ts` → `prisma:seed`); the runtime never touches the filesystem.

This is deliberately a document store, not the fully-normalized `missions`/`mission_activities`
tree envisioned for a later phase: it persists the exact authored content faithfully with a minimal,
reversible migration and no engine/domain redesign (RC-03 forbids new features/rewrites). The
Mission Engine reads the same rows via a pure mapper (`missionContentToEngineMission`).

## Impact Analysis

- **Tables:** 8 added (39 → 47). No existing table altered — the migration is additive only.
- **Data paths:** session write on lesson completion; lesson-plan upsert per planning; mission-
  progress upsert; template/history reads on the AI/admin paths; mission reads on the flow paths.
  All low volume.
- **Code:** ports unchanged. Composition roots (`daily-loop`, `ai`, `learning/mission`) swap to
  Prisma repos. `mission-loader.ts` becomes async DB reads (two pages `await` it). The in-memory
  classes are retained **only as test doubles** (annotated) except `InMemorySessionRepository`,
  which was unused after the swap and is **deleted** (dead code).
- **Rendering:** the admin segment is marked `force-dynamic` (it now reads live DB data and is
  permission-gated; it must not be statically prerendered). No UI/UX change.
- **Seed:** `prisma:seed` now also seeds the Mission Library (4 tracks × 10 missions) and the prompt
  templates (from `config/prompt-templates.ts`). Idempotent (upserts).
- **Ordering:** seed runs after `migrate deploy`; mission FK requires tracks before missions (the
  seed inserts a track then its missions).

## Migration Plan

- Forward: `prisma migrate deploy` applies `20260702010000_persistence_stores` (generated offline via
  `prisma migrate diff`, so it matches the schema exactly — drift-free), then `npm run prisma:seed`
  loads roles/permissions + A1 vocab + **Mission Library** + **prompt templates**.
- **Backfill:** none required — these are new stores. Existing users simply start writing sessions/
  progress/plans going forward; reads return empty until then (correct for a fresh store).

## Rollback

```sql
DROP TABLE IF EXISTS "ai_generation_jobs";
DROP TABLE IF EXISTS "prompt_versions";
DROP TABLE IF EXISTS "prompt_templates";
DROP TABLE IF EXISTS "mission_progress";
DROP TABLE IF EXISTS "content_missions";
DROP TABLE IF EXISTS "content_tracks";
DROP TABLE IF EXISTS "lesson_plans";
DROP TABLE IF EXISTS "learning_sessions";
```

Reversible. The code swap would also have to be reverted (re-wire the in-memory repos) if rolling
back the schema, since the runtime now depends on these tables.

## Alternatives

### A. Fully normalized mission tables (`missions`/`mission_activities`/`mission_exercises`/…)

The Learning Model V2 end-state. Rejected **for RC-03**: it is a larger migration + a mapping layer
in both directions, and RC-03 forbids rewrites. The document store persists the same content
faithfully now; normalization can follow as its own phase without changing the port or the schema
contract (the Zod schema stays the source of truth).

### B. Keep the Mission Library on the filesystem (read JSON at runtime)

Rejected: couples content to the deploy artifact, can't be edited without a redeploy, and the task
explicitly requires reading missions from the database (JSON only for seed).

### C. Gate the swap on `isDatabaseConfigured` (fall back to in-memory when no DB)

Rejected as the production posture: the runtime should depend on the database, consistent with the
already-Prisma repos (`PrismaReviewActivityRepository`, `PrismaLearnerProfileRepository`). The
in-memory classes remain only as **test doubles**. (The one retained no-op is the AI usage-log sink,
an intentional Milestone-1 decision so AI still runs with logging switched off — it is not an
in-memory store and is out of scope here.)

## Consequences

**Positive:** sessions/plans/progress survive restarts and scale across instances; the Mission
Library and prompt templates live in the DB (editable, seed-driven); no port/UI/UX change; additive,
reversible migration; the engine and the flow UI read the same authored content.

**Negative / Risks:** the runtime now hard-requires a database (no in-memory fallback) — pages that
read these stores fail without one, which is why this environment (no DB) verifies offline only.
Document columns (`data jsonb`) are validated in the app (Zod), not by DB constraints; a bad seed
would surface at read time. `SECURITY`/PII: none of these stores hold secrets or prompt text with
credentials.
